Decorator Pattern

חזרה לדף הבית

מטרת התבנית

תבנית Decorator מאפשרת להוסיף אחריות לאובייקט באופן דינמי, מבלי להשפיע על התנהגותם של אובייקטים אחרים מאותה מחלקה.

מתי להשתמש?

  • כאשר רוצים להוסיף אחריות לאובייקט באופן דינמי
  • כאשר רוצים להימנע מיצירת מחלקות רבות
  • כאשר רוצים להוסיף תכונות לאובייקטים קיימים
  • כאשר רוצים לשלב תכונות שונות

דוגמה בסיסית 

from abc import ABC, abstractmethod

# ממשק משותף
class Component(ABC):
    @abstractmethod
    def operation(self) -> str:
        pass

# רכיב בסיסי
class ConcreteComponent(Component):
    def operation(self) -> str:
        return "רכיב בסיסי"

# מחלקת בסיס למעטרים
class Decorator(Component):
    def __init__(self, component: Component):
        self.component = component

    def operation(self) -> str:
        return self.component.operation()

# מעטרים קונקרטיים
class ConcreteDecoratorA(Decorator):
    def operation(self) -> str:
        return f"מעטר A: {super().operation()}"

class ConcreteDecoratorB(Decorator):
    def operation(self) -> str:
        return f"מעטר B: {super().operation()}"

# שימוש
def client_code(component: Component):
    print(component.operation())

# יצירת אובייקטים
simple = ConcreteComponent()
decorated1 = ConcreteDecoratorA(simple)
decorated2 = ConcreteDecoratorB(decorated1)

# שימוש
client_code(simple)      # רכיב בסיסי
client_code(decorated1)  # מעטר A: רכיב בסיסי
client_code(decorated2)  # מעטר B: מעטר A: רכיב בסיסי

יתרונות

  • מאפשר הוספת אחריות באופן דינמי
  • מאפשר שילוב של תכונות שונות
  • מפחית את מספר המחלקות
  • מאפשר הרחבה של פונקציונליות

חסרונות

  • עלול ליצור אובייקטים מורכבים
  • עלול להקשות על הבנת הקוד
  • עלול ליצור ביצועים פחות טובים

השוואה לתבניות אחרות

תבנית הבדל
Adapter מתאים ממשקים לא תואמים, בעוד Decorator מוסיף אחריות
Composite משלב אובייקטים למבנה היררכי, בעוד Decorator מוסיף אחריות

תרגול

נסה ליישם את תבנית Decorator במקרים הבאים:

  1. הוספת תכונות לממשק משתמש
  2. הוספת פונקציונליות לקבצים
  3. הוספת תכונות למוצרים
