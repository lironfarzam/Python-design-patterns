Factory Method Pattern

חזרה לדף הבית

מהי תבנית Factory Method?

תבנית Factory Method היא תבנית עיצוב יצירתית המגדירה ממשק ליצירת אובייקט, אך מאפשרת למחלקות היורשות להחליט איזה מחלקה ליצור. התבנית מאפשרת דחיית יצירת האובייקט למחלקות היורשות.

הסבר למתחילים:

דמיינו שאתם רוצים ליצור מערכת שמטפלת בסוגים שונים של מסמכים (PDF, DOCX, TXT). במקום ליצור את האובייקטים ישירות, אתם משתמשים במחלקה בסיסית שמגדירה איך ליצור מסמך, ומחלקות יורשות שמחליטות איזה סוג מסמך ליצור.

הסבר למתקדמים:

התבנית מיישמת את עקרון ה-Open/Closed Principle (OCP) על ידי הפרדה בין יצירת האובייקט לשימוש בו. זה מאפשר הרחבה של המערכת מבלי לשנות את הקוד הקיים, ומאפשר החלפה דינמית של האובייקטים שיוצרו.

מתי להשתמש בתבנית?

למתחילים:

  • כאשר אתם לא יודעים מראש איזה סוג של אובייקט צריך ליצור
  • כאשר אתם רוצים לאפשר למחלקות יורשות להחליט איזה אובייקט ליצור
  • כאשר אתם רוצים להפריד בין יצירת האובייקט לשימוש בו

למתקדמים:

  • כאשר המערכת צריכה להיות עצמאית מאופן יצירת האובייקטים
  • כאשר יש מספר מחלקות שיכולות ליצור אובייקטים
  • כאשר רוצים לאפשר הרחבה של המערכת מבלי לשנות את הקוד הקיים
  • כאשר רוצים להסתיר את המימוש של האובייקט מהלקוח
  • כאשר יש צורך בתמיכה בסוגים שונים של אובייקטים
  • כאשר רוצים לאכוף קונסיסטנטיות ביצירת האובייקטים
  • כאשר יש צורך ליצור אובייקטים תואמים מבחינת נראות או התנהגות
  • כאשר רוצים להחליף בקלות בין סוגי אובייקטים בזמן ריצה

מבנה התבנית

הסבר בסיסי:

התבנית מורכבת משלושה חלקים עיקריים:

  • מחלקה בסיסית (Creator) - מגדירה את הממשק ליצירת האובייקט
  • מחלקות יורשות (ConcreteCreator) - מיישמות את יצירת האובייקט
  • אובייקטים (Product) - האובייקטים שיוצרים

הסבר מתקדם:

התבנית מורכבת מהמרכיבים הבאים:

  • Creator - מחלקה מופשטת המגדירה את הממשק ליצירת אובייקט
  • ConcreteCreator - מחלקה קונקרטית המיישמת את יצירת האובייקט
  • Product - ממשק המגדיר את הפעולות של האובייקט
  • ConcreteProduct - מימוש של הממשק המופשט, אובייקט ספציפי

היחסים בין המרכיבים:

  • Creator משתמש ב-Factory Method ליצירת האובייקט
  • ConcreteCreator מיישם את Factory Method ויוצר אובייקט קונקרטי
  • Product מגדיר את הממשק של האובייקט
  • ConcreteProduct מיישם את Product

דוגמה בסיסית

הסבר למתחילים:

בדוגמה הבאה, אנחנו יוצרים מערכת שמטפלת בסוגים שונים של מסמכים. המחלקה הבסיסית מגדירה איך ליצור מסמך, והמחלקות היורשות מיישמות את היצירה עבור כל סוג מסמך.

הסבר למתקדמים:

הדוגמה מדגימה את השימוש ב-Factory Method Pattern ליצירת אובייקטים שונים. הממשק המופשט מגדיר את הפעולות ליצירת האובייקט, והמימושים הקונקרטיים מספקים את המימוש הספציפי.

from abc import ABC, abstractmethod

class Document(ABC):
    @abstractmethod
    def open(self) -> str:
        pass

class PDFDocument(Document):
    def open(self) -> str:
        return "Opening PDF document"

class DOCXDocument(Document):
    def open(self) -> str:
        return "Opening DOCX document"

class TXTDocument(Document):
    def open(self) -> str:
        return "Opening TXT document"

class DocumentCreator(ABC):
    @abstractmethod
    def create_document(self) -> Document:
        pass
    
    def open_document(self) -> str:
        document = self.create_document()
        return document.open()

class PDFCreator(DocumentCreator):
    def create_document(self) -> Document:
        return PDFDocument()

class DOCXCreator(DocumentCreator):
    def create_document(self) -> Document:
        return DOCXDocument()

class TXTCreator(DocumentCreator):
    def create_document(self) -> Document:
        return TXTDocument()

# שימוש
if __name__ == "__main__":
    # יצירת מסמך PDF
    pdf_creator = PDFCreator()
    print(pdf_creator.open_document())
    
    # יצירת מסמך DOCX
    docx_creator = DOCXCreator()
    print(docx_creator.open_document())
    
    # יצירת מסמך TXT
    txt_creator = TXTCreator()
    print(txt_creator.open_document())

דוגמה מעשית - מערכת ניהול קבצים

הסבר למתחילים:

בדוגמה זו, אנחנו יוצרים מערכת ניהול קבצים שתומכת בסוגים שונים של קבצים. המחלקה הבסיסית מגדירה איך ליצור קובץ, והמחלקות היורשות מיישמות את היצירה עבור כל סוג קובץ.

הסבר למתקדמים:

הדוגמה מדגימה את השימוש ב-Factory Method Pattern ליצירת אובייקטים שונים. הממשק המופשט מגדיר את הפעולות ליצירת האובייקט, והמימושים הקונקרטיים מספקים את המימוש הספציפי.

from abc import ABC, abstractmethod
from datetime import datetime

class File(ABC):
    def __init__(self, name: str):
        self.name = name
        self.created_at = datetime.now()
    
    @abstractmethod
    def read(self) -> str:
        pass
    
    @abstractmethod
    def write(self, content: str) -> None:
        pass

class TextFile(File):
    def read(self) -> str:
        return f"Reading text file: {self.name}"
    
    def write(self, content: str) -> None:
        print(f"Writing to text file: {self.name}")

class BinaryFile(File):
    def read(self) -> str:
        return f"Reading binary file: {self.name}"
    
    def write(self, content: str) -> None:
        print(f"Writing to binary file: {self.name}")

class FileCreator(ABC):
    @abstractmethod
    def create_file(self, name: str) -> File:
        pass
    
    def process_file(self, name: str, content: str) -> None:
        file = self.create_file(name)
        file.write(content)
        print(file.read())

class TextFileCreator(FileCreator):
    def create_file(self, name: str) -> File:
        return TextFile(name)

class BinaryFileCreator(FileCreator):
    def create_file(self, name: str) -> File:
        return BinaryFile(name)

# שימוש
if __name__ == "__main__":
    # יצירת קובץ טקסט
    text_creator = TextFileCreator()
    text_creator.process_file("example.txt", "Hello, World!")
    
    # יצירת קובץ בינארי
    binary_creator = BinaryFileCreator()
    binary_creator.process_file("example.bin", "01010101")

יתרונות

למתחילים:

  • מאפשרת החלפה קלה של סוגי אובייקטים
  • מקלה על הוספת סוגי אובייקטים חדשים
  • מפרידה בין יצירת האובייקט לשימוש בו

למתקדמים:

  • מאפשרת הרחבה של המערכת מבלי לשנות את הקוד הקיים
  • מספקת ממשק אחיד ליצירת אובייקטים
  • מאפשרת החלפה דינמית של סוגי אובייקטים בזמן ריצה
  • מקלה על בדיקות (Testing) של המערכת
  • מאפשרת שימוש ב-Mock Objects לצורכי בדיקות

חסרונות

למתחילים:

  • עלולה ליצור מספר רב של מחלקות
  • עלולה להיות מורכבת מדי עבור מקרים פשוטים
  • מקשה על הוספת סוגי אובייקטים חדשים

למתקדמים:

  • עלולה להקשות על הרחבת הממשק
  • עלולה ליצור תלות בין המחלקות
  • עלולה להקשות על הבנת הקוד
  • עלולה ליצור קוד מיותר במקרים פשוטים

מלכודות נפוצות

למתחילים:

  • יצירת יותר מדי מחלקות קונקרטיות
  • שימוש בתבנית כאשר לא נדרשת
  • אי-הקפדה על התאמה בין האובייקטים

למתקדמים:

  • אי-שימוש בממשקים מופשטים
  • אי-הפרדה בין יצירת האובייקט לשימוש בו
  • אי-שימוש ב-Singleton עבור המחלקות הקונקרטיות

השוואה לתבניות אחרות

למתחילים:

תבנית הבדל
Abstract Factory מתמקדת ביצירת משפחות של מוצרים
Builder מתמקדת בבניית אובייקט מורכב

למתקדמים:

תבנית הבדל
Abstract Factory מתמקדת ביצירת משפחות של מוצרים, בעוד Factory Method מתמקדת ביצירת מוצר בודד
Builder מתמקדת בבניית אובייקט מורכב, בעוד Factory Method מתמקדת ביצירת אובייקט בודד
Prototype מתמקדת ביצירת אובייקטים על ידי שיבוט, בעוד Factory Method מתמקדת ביצירת אובייקטים חדשים
Singleton מתמקדת ביצירת מופע יחיד של מחלקה, בעוד Factory Method מתמקדת ביצירת אובייקטים חדשים

תרגול

למתחילים:

נסה ליישם את תבנית Factory Method במקרים הבאים:

  1. מערכת ניהול מסמכים עם תמיכה בפורמטים שונים
  2. מערכת ניהול קבצים עם תמיכה בסוגי קבצים שונים
  3. מערכת עיבוד תמונות עם תמיכה בפורמטים שונים

למתקדמים:

נסה ליישם את תבנית Factory Method במקרים הבאים:

  1. מערכת משחק עם דמויות שונות
  2. מערכת דיווח עם פורמטים שונים
  3. מערכת תקשורת עם פרוטוקולים שונים
  4. מערכת עיבוד תמונות עם פורמטים שונים

טיפים ליישום

למתחילים:

  • השתמש בממשקים מופשטים להגדרת המחלקות והאובייקטים
  • ודא שכל מחלקה יורשת מיישמת את הממשק המוגדר
  • הוסף בדיקות תקינות לאובייקטים שיוצרו

למתקדמים:

  • השתמש ב-Singleton עבור המחלקות הקונקרטיות אם נדרש
  • שקול שימוש ב-Registry או Configuration עבור בחירת המחלקה המתאימה
  • השתמש ב-Dependency Injection עבור הזרקת המחלקה
  • שקול שימוש ב-Abstract Factory בתוך Factory Method
  • הוסף תיעוד מפורט לממשקים והמימושים
